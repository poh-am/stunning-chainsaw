## Purpose
Guide Business Analysts through a structured requirements gathering process to ensure completeness and clarity.

## Stakeholder Identification
- [ ] Identify all relevant stakeholders
- [ ] Determine decision-makers vs. contributors
- [ ] Map stakeholder communication preferences
- [ ] Schedule initial interviews or workshops

## Requirements Elicitation Checklist
- [ ] Business objectives and success metrics defined
- [ ] Current state (as-is) process documented
- [ ] Future state (to-be) process outlined
- [ ] User roles and personas identified
- [ ] Use cases and user scenarios captured
- [ ] Business rules and constraints documented
- [ ] Data requirements and sources identified
- [ ] Integration points mapped
- [ ] Non-functional requirements gathered (performance, security, compliance)

## Requirements Documentation Checklist
- [ ] Requirements traced to business objectives
- [ ] Each requirement has a unique identifier
- [ ] Acceptance criteria defined for each requirement
- [ ] Requirements prioritized (MoSCoW or similar)
- [ ] Dependencies and risks documented
- [ ] Assumptions and constraints listed
- [ ] Glossary of terms included

## Requirements Validation Checklist
- [ ] Requirements reviewed with stakeholders
- [ ] Conflicts or gaps identified and resolved
- [ ] Feasibility confirmed with technical team
- [ ] Sign-off obtained from Product Manager
- [ ] Baseline established for change control

## Requirement Document Template

### Requirement ID: [REQ-XXX]
- **Title**: [Short descriptive title]
- **Description**: [Detailed requirement statement]
- **Business Justification**: [Why is this needed?]
- **Acceptance Criteria**:
  - Given [precondition]
  - When [action]
  - Then [expected result]
- **Priority**: [Must/Should/Could/Won't]
- **Source**: [Stakeholder or document reference]
- **Dependencies**: [Related requirements or systems]
- **Status**: [Draft/Review/Approved/Implemented]
